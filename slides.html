The Problem


AI agents can run ML experiments autonomously.

But why would anyone donate their GPU?


We need an incentive mechanism that rewards agents proportional to their research contributions.

"AI agents can run ML experiments autonomously"
Coding agents like Claude Code can already modify training scripts, run training, check metrics, and iterate — all without human input. This is a solved problem.
"why would anyone donate their GPU?"
GPUs are expensive ($2-3/hr for an H100). Without compensation, only hobbyists or altruists would contribute. This limits the scale of collaborative research.
"incentive mechanism"
An economic system where people get paid proportionally to how much they improve the model. This is what Bittensor provides — it's a blockchain designed specifically for rewarding useful computational work.
"rewards agents proportional to their research contributions"
Not flat payment — the better your experiment result (lower val_bpb), the more TAO you earn. This creates competitive pressure to actually do good science.

What is Autoresearch?


Give an AI agent a small LLM and let it experiment autonomously.


Modify code → Train → Check val_bpb → Keep or discard → Repeat


"The next step for autoresearch is that it has to be asynchronously massively collaborative for agents (think: SETI@home style)." — @karpathy, March 2026

"Give an AI agent a small LLM"
We give the agent a small language model (e.g. ~50M parameters) — small enough to train in minutes, but big enough that improvements are meaningful and transferable to larger models.
"let it experiment autonomously"
The AI agent (Claude Code) reads the training script, decides what to change (architecture, learning rate, optimizer, etc.), edits the code, runs training, and evaluates — all on its own, no human in the loop.
"Modify code → Train → Check val_bpb → Keep or discard → Repeat"
The core loop. val_bpb = validation bits-per-byte. It measures how many bits the model needs to encode each byte of unseen text. Lower = better predictions. If the change improves val_bpb, keep it. If not, revert and try something else.
"SETI@home style" — Karpathy quote
SETI@home was a project where millions of people donated idle CPU time to search for alien signals. Karpathy's vision: do the same thing but for ML research — thousands of agents on different machines, all experimenting in parallel and sharing results.

The Experiment Loop


  1. THINK — check what the swarm tried, pull global best
  2. CLAIM — register experiment to avoid duplicates
  3. EDIT — modify train.py (arch, optimizer, hyperparams)
  4. TRAIN — run a fixed-budget experiment
  5. PUBLISH — share results + insights with the swarm
  6. LOOP — repeat forever
"THINK — check what the swarm tried, pull global best"
Before doing anything, the agent reads the shared state: what experiments have already been run, which ones worked, what insights others posted. It pulls the current best training config as a starting point. This avoids wasting time on things already tried.
"CLAIM — register experiment to avoid duplicates"
The agent announces "I'm about to try X" in the shared coordination layer. Claims expire after 15 minutes. This prevents two agents from running the same experiment simultaneously.
"EDIT — modify train.py (arch, optimizer, hyperparams)"
The agent edits the actual Python training script. This could mean changing the number of layers, switching optimizers, adjusting learning rate schedules, modifying attention patterns — any code-level change to the model or training procedure.
"TRAIN — run a fixed-budget experiment"
Each experiment gets a fixed compute budget (5 min on H100, 30s in demo). This ensures fair comparison — every experiment uses the same amount of compute, so improvements come from better ideas, not more GPU time.
"PUBLISH — share results + insights with the swarm"
After training, the agent shares: the val_bpb result, the full training code, and qualitative insights ("increasing depth helped because..."). Other agents can learn from this.
"LOOP — repeat forever"
The agent goes back to step 1 and keeps going indefinitely. Our overnight run did 83 iterations over ~7 hours.

"Params ~50M / ~0.3M"
Number of trainable weights in the neural network. 50M is small by modern standards (GPT-2 was 1.5B) but large enough for meaningful experiments. 0.3M is tiny — just for demo purposes.
"Layers / Heads"
Transformer architecture. Layers = how many transformer blocks stacked. Heads = parallel attention patterns in each layer. More layers = deeper reasoning. More heads = attending to different parts of the input simultaneously.
"Embed dim"
The size of each token's internal representation vector. 512 means each token is a 512-dimensional vector inside the model. Bigger = more capacity but more memory/compute.
"Vocab: 8192 BPE / ~69 chars"
BPE (Byte Pair Encoding) is a subword tokenizer — splits text into common subword chunks. 8192 tokens in production. Demo uses raw characters (~69 unique chars in Shakespeare).
"FA3 + sliding"
Flash Attention 3 — a GPU-optimized attention algorithm that's 2-4x faster than standard attention. "Sliding" = sliding window attention, where tokens only attend to nearby tokens, not the entire sequence. Saves memory.
"Muon + AdamW"
Muon is a newer optimizer that uses momentum in a specific way that outperforms AdamW on small models. We use Muon for most parameters and AdamW specifically for embeddings (which Muon doesn't handle well).
"SDPA"
Scaled Dot-Product Attention — PyTorch's built-in efficient attention. Not as fast as Flash Attention but works on CPU and Apple MPS (Mac GPUs).
"Time budget: 5 min / 30s"
Fixed wall-clock time for each experiment. This is the key constraint — everyone gets the same compute budget, so improvements come from smarter training, not longer training.
"tinyshakespeare"
A classic ML benchmark dataset — ~1MB of Shakespeare text. Small enough for quick experiments, complex enough to be non-trivial.

Each experiment: 5 min train + think + publish. Agent ran ~7 hours total.

"Halve batch 524K→262K (more steps)"
Smaller batch size = each gradient update sees fewer tokens, but you get more updates in the same time budget. More updates = more opportunities for the model to learn. This is often the single biggest lever in time-constrained training.
"Warmdown 0.5→0.7 (more cooldown)"
Warmdown = the fraction of training spent reducing the learning rate toward zero. Going from 0.5 to 0.7 means 70% of training is spent cooling down. This lets the model settle into a better optimum.
"Add 5% warmup"
Warmup = gradually increasing the learning rate at the start instead of jumping to full speed. Prevents early instability when the model's weights are still random.
"Depth 9, aspect_ratio 57"
Making the model deeper (9 layers instead of 8) with a different width-to-depth ratio. Deeper models can learn more complex patterns but need careful tuning to train stably.
"x0_lambda init 0.1→0.05"
A parameter controlling the initial scale of residual connections. Smaller = the residual branches contribute less at the start, which can help training stability in deeper models.
"Window patterns, embedding LR, RoPE base freq"
More nuanced changes: sliding window attention sizes, separate learning rate for embedding layers (which often benefit from a different rate), and RoPE (Rotary Position Embedding) base frequency which controls how the model encodes token positions.
"Random seed 42→137"
Changing the random seed changes weight initialization and data shuffling order. The fact that this helped (0.977 vs previous best) shows the loss landscape has multiple basins and initialization matters even at this scale.

"SETI@home for ML research"
SETI@home let millions donate idle CPU cycles to search for alien signals. We do the same but for ML — people donate GPU cycles to run AI experiments. The difference: our contributors actually earn money (TAO).
"Multiple AI agents on different GPUs, collaborating via Ensue"
Ensue is a real-time state sync platform (like a shared document but for structured data). Each agent reads/writes to a shared namespace so they can see what others are doing, what's been tried, and what worked.
"claims/<agent>--<slug>"
When an agent starts an experiment, it writes a claim: "I'm working on X." Claims expire after 15 minutes. If an agent crashes, its claim auto-expires and someone else can try that idea.
"results/<agent>--<slug>"
Completed experiments are stored here with the full val_bpb score, memory usage, and the actual train.py code that produced the result. Other agents can read this to learn what worked.
"hypotheses/<agent>--<slug>"
Ideas for experiments that an agent thought of but didn't have time to run. Other agents can pick these up — it's like a shared "to try" list.
"insights/<agent>--<slug>"
Qualitative learnings: "larger batch size hurt because we get fewer steps in the time budget" or "the model benefits from deeper architectures." This is what makes the swarm collectively intelligent — they share reasoning, not just numbers.
"best/train_py" and "best/tier/<tier>/"
The globally best training script, and per-hardware-tier bests. When an agent starts a new experiment, it pulls the best config for its tier as a baseline, then tries to improve on it.

"Fair comparison across hardware"
A model config optimized for 80GB H100 would crash on a 12GB RTX 4070. We can't compare results across different hardware classes, so we tier them.
"small ≤16 GB"
Consumer-grade GPUs. Can train models up to ~10-15M parameters with typical batch sizes. Examples: RTX 4070, RTX 3060, RTX 4060.
"medium ≤24 GB"
High-end consumer GPUs. Can handle ~30-50M parameter models. Examples: RTX 3090, RTX 4090 — the "prosumer" tier.
"large ≤48 GB"
Professional/workstation GPUs. Can handle larger models and batch sizes. Examples: A6000, L40 — typically found in research labs.
"xl >48 GB"
Data center GPUs with massive VRAM. Examples: A100 (40/80GB), H100 (80GB), H200 (141GB). These are what cloud providers offer.
"Agents pull configs from their own tier so they don't OOM"
OOM = Out Of Memory — the GPU runs out of VRAM and the training crashes. By pulling the best config for your specific tier, you get a config that's guaranteed to fit in your GPU's memory.

"coord.post_insight(...) / coord.publish_hypothesis(...)"

Turn the collaborative research swarm into a Bittensor subnet.

"Before (Altruistic) — Agents donate GPU time"
In Autoresearch@Home without Bittensor, people run agents out of goodwill. No compensation. This is like early open-source: works for enthusiasts, doesn't scale.
"No incentive to participate"
GPU time costs real money. An H100 is $2-3/hour on cloud. Without payment, only researchers with grant money or hobbyists would contribute.
"Trust-based collaboration"
Anyone could submit fake results or copy others' work. There's no enforcement mechanism — you just trust that participants are honest.
"After (Incentivized) — Agents earn TAO"
TAO is Bittensor's native cryptocurrency. It has real monetary value. Miners earn TAO proportional to the quality of their work — the better your experiments, the more you earn.
"Better results = more rewards"
The scoring formula (best/yours)² means the best miner earns the most. A miner 5% worse than the best earns significantly less. This creates competitive pressure to innovate.
"Anti-cheat built in"
Validators enforce rules: dedup (can't copy someone's code), staleness (must submit fresh results), sanity bounds (can't fake impossibly good metrics). Cheating is economically unprofitable.
"Bittensor subnet"
Bittensor is a network of ~50+ subnets, each doing different work (text gen, image gen, storage, etc.). Each subnet has miners (do work) and validators (score work). The blockchain distributes TAO rewards. Our subnet's "work" is ML research.

"MINER (HTTP :8091)"
Each miner is a Python HTTP server running on port 8091. Anyone with a GPU can run one. In production, this would be a Bittensor axon (their built-in server framework).
"each GPU participant"
One miner per GPU. If you have 4 GPUs, you run 4 miners on different ports. Each one independently runs experiments.
"Axon — /run endpoint"
"Axon" is Bittensor's name for a miner's server endpoint. When the validator sends POST /run, the miner spawns train_lite.py, waits for it to finish, and returns the result.
"VALIDATOR (HTTP :8092) — single instance"
There's one validator that orchestrates everything. It queries all miners, scores their results, and sets weights on the blockchain. In production Bittensor, there can be multiple validators.
"Validator loop — Queries miners every N seconds"
Every 45 seconds, the validator sends POST /run to every registered miner. This triggers a new experiment round.
"Scoring — Anti-cheat checks — set_weights() (mock)"
The validator checks results for cheating, scores them using (best/theirs)², and calls set_weights(). "(mock)" means in our demo we log what would be sent to the blockchain instead of actually sending it.
"SUBTENSOR — Yuma Consensus → TAO rewards ∝ weights"
Subtensor is Bittensor's blockchain. Yuma Consensus is their consensus algorithm that aggregates validator weights and distributes TAO. "∝ weights" means rewards are proportional to your weight — higher weight = more TAO.

"Validator Loop (every 45s)"
The validator wakes up every 45 seconds and triggers a new round of experiments. 45s is long enough for a 30s demo training run plus overhead.
"POST /run to each miner"
HTTP POST request telling the miner "run an experiment now." The miner is a web server — the validator is the client.
"Miner runs train_lite.py (~30s)"
The miner spawns train_lite.py as a subprocess with a time budget. The script trains a small GPT model, prints metrics to stdout, and exits.
"returns ExperimentResult as JSON"
The miner parses the training output, packages it into an ExperimentResult dataclass, serializes to JSON, and returns it as the HTTP response.
"SubmissionStore — anti-cheat + dedup + scoring"
A class inside the validator that stores all submissions and enforces rules: no duplicate code hashes, no stale results, no impossible metrics, only completed experiments score.
"Weights: score = (best / miner_bpb)²"
The scoring formula. If the best val_bpb is 2.80 and yours is 2.85, your score = (2.80/2.85)² = 0.965. The squaring amplifies differences. Scores are normalized so all weights sum to 1.0.
"TAO Rewards ∝ weight"
The Bittensor blockchain distributes TAO tokens every epoch (~12 min). Your share of the TAO is proportional to your weight. Weight 0.5 out of 1.0 = you get 50% of the subnet's TAO emissions.

@dataclass
class ExperimentResult:

    val_bpb: float = 0.0          # the key metric

Demo: plain dataclass over HTTP/JSON. Production: bt.Synapse over dendrite/axon.

"@dataclass class ExperimentResult"
A Python dataclass — a clean way to define a structured data object with typed fields. This is the contract between miner and validator: every experiment returns this exact shape of data.
"agent_id: str"
Unique identifier for the miner (e.g. "raven", "nova"). Used for the leaderboard and to track who submitted what.
"val_bpb: float — the key metric"
Validation bits-per-byte. THE metric that determines your score and TAO earnings. It measures the model's prediction quality on text it wasn't trained on. Lower = better. Typical values: 2.8 for demo, 0.98 for H100 production.
"memory_gb: float"
Peak GPU memory used during training. Tracked for VRAM tier fairness and to detect anomalies.
"training_seconds / total_tokens_M / num_steps / num_params_M"
Metadata about the training run: how long it took, how many tokens were processed (in millions), how many gradient steps, model size. Used for analytics and anomaly detection.
"train_py_hash: str — SHA256 for dedup"
The SHA256 hash of the training script's source code. If two miners submit the same hash, only the first gets credit. This prevents copying someone else's successful code.
"accepted: bool / score: float — set by validator"
The miner doesn't set these — the validator fills them in after anti-cheat checks and scoring. This ensures miners can't self-score.
"bt.Synapse over dendrite/axon"
In production Bittensor: bt.Synapse is their typed message class (like our dataclass but with built-in serialization). Dendrite = client (validator), Axon = server (miner). Handles auth, routing, and encryption natively.

The Miner


HTTP server that runs experiments on demand.

Validator sends POST /run → miner runs train_lite.py

Parses output, returns ExperimentResult as JSON.


One experiment at a time (lock-guarded). GET /status shows idle/running.

"HTTP server that runs experiments on demand"
The miner is literally a Python web server (http.server). It sits idle waiting for requests, just like any web server — but instead of serving web pages, it runs ML experiments.
"Validator sends POST /run → miner runs train_lite.py"
POST /run is the trigger. The validator acts as a client sending HTTP requests. The miner receives the request, spawns the training script as a subprocess, waits for it to finish, and returns the result as JSON.
"Parses output, returns ExperimentResult as JSON"
train_lite.py prints metrics to stdout (like "val_bpb=2.834"). The miner parses these strings, constructs an ExperimentResult object, converts it to JSON, and sends it back in the HTTP response body.
"One experiment at a time (lock-guarded)"
A threading.Lock ensures only one experiment runs at once. If the validator sends a second request while one is running, it waits. This prevents GPU memory conflicts.
"GET /status shows idle/running"
A monitoring endpoint. The UI and CLI dashboard can poll this to show whether the miner is currently training or waiting for the next request.

Miner Internals


def _run_experiment(self) -> ExperimentResult:
    """Run train_lite.py and return the parsed result."""
    env["AUTORESEARCH_TIME_BUDGET"] = str(self.server.time_budget)

    proc = subprocess.run(
        [sys.executable, "train_lite.py"],
        capture_output=True, text=True,
        timeout=self.server.time_budget + 60,
        env=env,
    )

    metrics = parse_train_output(proc.stdout)
    return ExperimentResult(
        agent_id=miner_id,
        val_bpb=metrics["val_bpb"],
        train_py_hash=hash_file("train_lite.py"),
        status="completed",
        ...
    )
"env['AUTORESEARCH_TIME_BUDGET'] = str(self.server.time_budget)"
The time budget is passed as an environment variable to the training script. train_lite.py reads this and knows how long it's allowed to train (30s in demo). This keeps the experiment standardized.
"subprocess.run([sys.executable, 'train_lite.py'], ...)"
subprocess.run spawns a completely separate Python process. capture_output=True captures stdout/stderr. text=True returns strings instead of bytes. This isolates the training from the miner server — if training crashes, the miner keeps running.
"timeout=self.server.time_budget + 60"
The +60 second buffer gives the training script time to finish cleanup after its internal budget expires. Without this, the subprocess might get killed before it can print final metrics.
"metrics = parse_train_output(proc.stdout)"
Parses the raw text output of train_lite.py looking for key-value pairs like "val_bpb=2.834", "peak_memory_gb=1.2", etc. Extracts them into a dictionary.
"train_py_hash=hash_file('train_lite.py')"
Computes SHA256 of the training script's source code. This is the dedup key — if another miner submits a result from identical code, only the first submission counts.

"Two responsibilities"
The validator does two things: (1) actively queries miners and scores results, and (2) passively serves data to the UI via a REST API. Both run in the same process.
"Query Loop — POST /run to each miner"
Every 45 seconds, the validator loops through all registered miner URLs and sends POST /run to each one. This triggers experiments.
"Collect ExperimentResults"
Gather the JSON responses from each miner. Failed requests (timeouts, network errors) are logged and the miner is skipped for that round.
"Anti-cheat + store"
The SubmissionStore validates each result (dedup, staleness, sanity, status) and stores accepted ones. Rejected results are still logged in the feed for transparency.
"Print leaderboard + weights"
After each round, the validator prints a ranked leaderboard to the terminal and computes the weights that would be sent to the blockchain.
"/api/leaderboard"
Returns JSON: ranked list of miners with their best val_bpb, score, weight, experiment count. The web UI and CLI dashboard poll this.
"/api/feed"
Chronological stream of ALL experiment results — both accepted and rejected. Shows the full history of the subnet.
"/api/results"
Raw result data optimized for charting — val_bpb over time for each miner. The UI uses this for the line chart.
"/api/status"
Subnet health: number of active miners, total experiments, best val_bpb, uptime. Quick health check.

Scoring Formula


Lower val_bpb = better model = higher score


Best miner: score = 1.0

Other miners: score = (best / miner_bpb)²


Scores normalized to weights that sum to 1.0

"Lower val_bpb = better model = higher score"
val_bpb (bits-per-byte) is an information-theoretic metric. A perfect model that knows exactly what comes next would need 0 bits. A random guesser over 256 bytes would need 8 bits. Lower = more accurate predictions = better language model.
"Best miner: score = 1.0"
The miner with the lowest (best) val_bpb gets the maximum score of 1.0. This is the reference point for everyone else.
"Other miners: score = (best / miner_bpb)²"
Example: if best = 2.80 and yours = 2.85, your score = (2.80/2.85)² = 0.965. The squaring is deliberate — it amplifies small differences. A 2% worse val_bpb gives ~4% lower score. This creates strong incentive to push for every improvement.
"Scores normalized to weights that sum to 1.0"
If there are 3 miners with scores [1.0, 0.96, 0.90], total = 2.86, so weights = [0.350, 0.336, 0.315]. These weights go directly to the blockchain. TAO distribution follows these weights exactly.

Prevents copying others' work, submitting stale results, or gaming metrics

"Dedup — train_py_hash — first submitter wins"
Every miner's training script is SHA256-hashed. If miner B copies miner A's code and submits the same hash, miner B's submission is rejected. Only the first miner to submit a given hash gets credit. This prevents free-riding.
"Staleness — Results older than 1 hour are ignored"
The timestamp in ExperimentResult must be within the last hour. This prevents miners from saving a good result and replaying it forever without doing new work.
"Sanity bound — val_bpb ≥ 0.5"
A model training for only 5 minutes on a small dataset physically cannot achieve val_bpb below ~0.5 bits per byte. Any submission claiming lower is fabricated. This catches miners who hardcode fake metrics.
"Status check — Only 'completed' results score"
If training crashed, timed out, or errored, the status field will be 'failed' instead of 'completed'. Failed experiments get zero score. This prevents miners from submitting partial/garbage results.

  nova    weight=0.4970  →  TAO reward ∝ 0.4970
"=== ROUND 1 ==="
Each round = one cycle of the validator querying all miners. Rounds happen every 45 seconds.
"[raven] requesting experiment..."
The validator sends POST /run to miner "raven". Raven spawns train_lite.py and starts training the model.
"[raven] val_bpb=2.8341 — ACCEPTED (score=1.000)"
Raven finished training: its model achieved val_bpb = 2.8341 bits per byte on the validation set. Since this is the best score, raven gets score 1.0. "ACCEPTED" means it passed all anti-cheat checks.
"[nova] val_bpb=2.8512 — ACCEPTED (score=0.988)"
Nova got a slightly worse val_bpb (2.8512). Its score = (2.8341/2.8512)² = 0.988. Still close — the difference is only ~0.6% in val_bpb.
"--- Leaderboard ---"
After each round, the validator prints a ranked list of all miners. #1 is the best. Shows val_bpb, score, and total experiments completed.
"--- Setting Weights (mock) ---"
"mock" = we're simulating the blockchain call. In production, this would actually call set_weights() on the Bittensor subtensor blockchain.
"raven weight=0.5030 → TAO reward ∝ 0.5030"
Raven's weight: its score (1.0) divided by the sum of all scores. This means raven would earn ~50.3% of the subnet's TAO emissions. The weights are nearly equal because the miners' results are very close.

"Karpathy's Autoresearch — Single agent, single GPU"
Stage 1: Andrej Karpathy demonstrated that a single AI agent on one GPU can autonomously improve a model by modifying code, training, evaluating, and iterating. Powerful proof of concept, but limited to one machine.
"Autoresearch@Home — Multiple agents via Ensue (altruistic)"
Stage 2: We extended this to multiple agents on different GPUs, sharing their work via the Ensue coordination platform. The swarm collaborates and gets smarter together. But it's altruistic — no one gets paid, so it doesn't scale beyond enthusiasts.
"Autoresearch TAO — Incentivized subnet on Bittensor"
Stage 3 (what we built): same multi-agent research swarm, but wrapped as a Bittensor subnet. Now agents earn TAO tokens. The economic incentive transforms this from a hobby into a self-sustaining network where anyone with a GPU can participate and get paid.
"Agents earn TAO proportional to val_bpb improvements"
The key economic design: your reward is directly proportional to how much you improve the model. score = (best/yours)², normalized to weights, TAO distributed per weight. Better science = more money.
